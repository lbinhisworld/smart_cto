# 数字化问题跟进阶段设计

> 最近更新：2025-03-18  
> 变更：**过程日志时间线备注统一为三项**——所有时间线内容卡片备注均为：**耗时，输入 token，输出 token**（顺序一致）。（1）**输入卡片**：耗时为 0，输出 token 为 0，输入 token 为**后续**大模型调用返回的 prompt_tokens；（2）**输出/确认卡片**：耗时为大模型返回耗时，输入 token 为 0，输出 token 为大模型返回的 completion_tokens；（3）**压缩卡片**：耗时为该次大模型返回耗时，输入 token 为该次调用的 prompt_tokens，输出 token 为该次调用的 completion_tokens。顶部汇总与任务标题行按上述规则累计。**局部 ITGap 流程**——确认所有输出后先推送**压缩确认块**（「我即将开始对局部 ITGap 分析做上下文压缩…」+ 确认）；用户点击确认后再按环节执行压缩，每完成一环向 task9 时间线推送「压缩」块；**全部压缩完成后**再推送**任务完成确认块**（「是否确认局部 ITGap 分析任务已经完成？」+ 确认）；用户点击确认后推送「任务完成」、推进状态并切换当前任务为角色与权限模型推演（task10）。**重启当前**——按钮采用事件委托绑定，无当前任务（全部已完成）时提示「当前没有可重启的任务」。  
> 历史：2025-03-17 局部 ITGap 压缩/任务完成确认流程；问题详情刷新行为、角色与权限页面、沟通历史与过程日志（722px、输入/输出 token 分列）；端到端流程绘制、任务完工确认、Ask 模式等。

## 0. 状态管理模块（每个数字化问题单）

### 0.1 15 种任务状态（即任务名称）

每个数字化问题单均包含以下 15 种状态，每种状态对应一个任务名称，**依次顺序触发**：

| 序号 | 任务 ID | 任务名称（状态名） |
|------|---------|--------------------|
| ① | task1 | 企业背景洞察 |
| ② | task2 | 商业画布加载 |
| ③ | task3 | 需求逻辑构建 |
| ④ | task4 | 绘制价值流 |
| ⑤ | task5 | IT现状标注 |
| ⑥ | task6 | 痛点标注 |
| ⑦ | task7 | 端到端流程绘制 |
| ⑧ | task8 | 全局 ITGap 分析 |
| ⑨ | task9 | 局部 ITGap 分析 |
| ⑩ | task10 | 角色与权限模型推演 |
| ⑪ | task11 | 核心业务对象推演 |
| ⑫ | task12 | 全局架构设计 |

| ⑭ | task14 | 链条串联与闭环 |
| ⑮ | task15 | 价值回溯与自检 |

### 0.2 顺序触发与当前任务追踪

- **顺序触发**：状态①结束后，当前任务切换到状态②；状态②结束后，当前任务切换到状态③；依次类推。
- **当前任务**：系统对每个问题单维护一个「当前任务状态」，表示当前应执行或正在执行的任务。当前任务 = 下一个待完成的状态（第一个未完成的 task 即为当前任务）。
- **状态推进**：任一任务完成（满足评价标准并用户确认）后，当前任务自动指向下一个未完成任务；若所有 15 个任务均已完成，则当前任务为「已全部完成」。

### 0.3 任务扩展字段（在原有任务字段基础上）

每个任务（或每个问题单下针对每个任务的状态记录）增加如下字段：

| 字段 | 说明 |
|------|------|
| **输入** | 该任务所需的前置输入（如：企业基本信息、BMC、需求逻辑、价值流图等），用于描述本任务的触发前提 |
| **动作** | 该任务的核心动作描述（如：提炼企业信息、生成 BMC、绘制价值流、标注 IT 现状等） |
| **输出反馈** | 该任务完成后的输出物及用户反馈形式（如：结构化 JSON、Markdown 卡片、用户确认等） |
| **任务状态** | 本任务在此问题单下的状态：`pending`（待执行）、`in_progress`（进行中）、`completed`（已完成） |

### 0.5 与既有存储的映射

- **currentTaskIndex / currentTaskId**：可由既有 `completedStages`、`workflowAlignCompletedStages`、`itGapCompletedStages`、`currentMajorStage` 等推导出「当前任务」为 task1–task15 中的哪一个。
- **任务状态（task status）**：已完成任务对应 `completed`；当前正在执行的任务对应 `in_progress`；尚未到达的任务对应 `pending`。
- **输入 / 动作 / 输出反馈**：可从现有任务定义（objective、evaluationCriteria）及触发事件中提炼，用于任务卡片或工作区展示。

### 0.6 沟通历史中任务扩展字段的展示（已实现）

在问题详情页「沟通历史」侧滑面板中，每个任务为可折叠卡片。展开后包含**两个 Tab**：「任务详情」与「过程日志」。

- **任务详情 Tab**：展示归属阶段、任务目标、评估标准、**输入**（该任务的前置输入说明）、**动作**（核心动作描述）、**输出反馈**（完成后的输出物及用户反馈形式）、**任务状态**（由当前问题单推导：`已完成` / `进行中` / `待执行`）。任务过程日志仅从既有聊天记录提取，不再写入新条目（相关 Log 类写入已移除）。
- **过程日志 Tab**：以时间线展示该任务下的沟通记录。**沟通历史面板**：宽度 722px，底色深紫（#1a0a2e）；**顶部汇总**为三项（从左到右）：**总耗时**（分秒显示：≥1 分钟为「X分Y秒」，&lt;1 分钟为「Y秒」）、**输入 token**、**输出 token**（千分位），由所有任务时间线数据汇总；其中「输入 token」只累计标签为「输入」的条目的 prompt_tokens，「输出 token」只累计标签为「输出」与「确认」的条目的 completion_tokens（同一内容块由输出变为确认只计一次），总耗时只累计非「输入」条目的大模型耗时。**任务标题行**：任务名称右侧为本任务的**耗时、输入 token、输出 token** 三列（与下方时间线内容块对齐）。每条时间线内容块的**标题栏由时间戳 + 日志类型标签**构成，部分类型可在标签右侧显示**备注**（如环节名称、上下文来源、session 计划等）；标题栏最右侧为**三项统计备注**（**耗时**（白色、秒）、**输入 token**、**输出 token**），顺序一致。**输入卡片**：耗时=0、输出 token=0、输入 token=后续一条消息的 llmMeta.usage.prompt_tokens；**输出/确认卡片**：耗时=该条 durationMs、输入 token=0、输出 token=该条 completion_tokens；**压缩卡片**：耗时=该条 durationMs、输入/输出 token=该条 prompt_tokens/completion_tokens。数据来自消息 `llmMeta`（usage、durationMs）。日志类型为：**输入**（用户发言）、**输出**（大模型提炼输出）、**确认**（系统确认等）、**修正**（用户对提炼内容给出修改意见并发送后）、**讨论**（意图为讨论或系统提炼）、**上下文**（意图为查询或执行的意图提炼卡片）、**任务完成**（用户点击任务完工确认「已完成」后写入）。用户点击「修正」时**不**向过程日志推送内容；用户点击「已完成」后，仅在过程日志中追加「任务完成」类型条目，**不在聊天区展示**。**输入**类型条目在时间线详情中**仅展示用户输入的正文**，不展示整段聊天内容块 JSON。**任务完成**类型条目按任务区分展示内容（如端到端流程绘制为「用户确认端到端流程绘制」、全局 ITGap 分析为「用户确认全局 ITGap 分析」等），标签为「任务完成」。**任务完成**类条目按消息的 `taskId` 归入对应任务时间线。**端到端流程 JSON 块**（e2eFlowJsonBlock）：聊天区推送该块时**立即**在过程日志中增加一条时间线，标签为「输出」、包含该 JSON 数据；用户点击卡片「确认」后该条标签变为「确认」；存在未确认的端到端流程 JSON 块时，沟通历史中 task7 显示为「进行中」、当前任务不提前变为 task8。**局部 ITGap 分析过程日志**：提取内容/确认类时间线在标签右侧显示对应**环节名称**（stepNameForHead）；用户确认开始局部 ITGap（生成 session 列表）时，向 task9 过程日志推送两条**独立**「上下文」内容块：**「价值流图 json」**、**「全局 ITGap 分析 json」**（`localItGapContextBlock`，标签旁备注为 contextLabel），**不向聊天窗口推送**；用户点击 session 确认后推送的条目在「确认」标签旁备注**「ITGap 分析 session 计划」**（sessionPlanNoteForHead）；全部环节确认后先推送压缩确认块，用户确认后再按环节依次推送标签为**「压缩」**的内容块（标签后带环节名，展示耗时、token 及压缩后 JSON），全部压缩完成后再推送任务完成确认块，用户确认后写入「任务完成」并切换当前任务为 task10。**全局 ITGap 分析**：任务上下文内容卡片（taskContextBlock，task8）**仅写入沟通历史过程日志**，**不在聊天区展示**。**task9**：点击局部 ITGap 分析任务按钮时**不**推送 taskContextBlock（避免无标题备注的「上下文」条）。过程日志时间线标题栏不换行（`white-space: nowrap`），过长时标题栏可横向滚动。

**任务标题栏显示**：已完成任务的标题栏字体为**绿色**；进行中为**蓝色**；待执行沿用默认文本色。CSS 类：`problem-detail-history-task-done`、`problem-detail-history-task-current`。

数据来源：`TASK_EXTRA_FIELDS`（js/config.js）为每个 taskId 提供 input、action、outputFeedback；任务状态由 `isTaskCompleted` 及「第一个未完成任务」逻辑计算得出。

## 2. 任务目标设计

| 任务 | 目标描述 |
|------|----------|
| **企业背景洞察** | 基于客户输入的企业信息，提炼并结构化企业基本信息，为后续 BMC 与需求逻辑分析提供基础。 |
| **商业画布加载** | 基于企业基本信息，运用商业模式画布（BMC）框架生成结构化商业分析，识别客户细分、价值主张、渠道通路等九大模块。 |
| **需求逻辑构建** | 基于客户初步需求、企业基本信息、BMC 三个维度，产出需求背后的逻辑链条分析，明确行业特性、商业模式局限与业务痛点的因果关联。 |
| **绘制价值流** | 基于 enterprise_info、bmc_data、requirement_logic 生成业务核心价值流图，划分阶段与环节，标注执行角色与预估耗时。 |
| **IT 现状标注** | 结合需求逻辑，在价值流图每个环节节点标注该环节的 IT 支撑方式（手工/系统），区分纸质、excel 或具体系统名称。 |
| **痛点标注** | 结合需求逻辑，在价值流图每个环节节点提炼该环节涉及到的痛点，为后续 IT Gap 分析提供输入。 |
| **端到端流程绘制** | 从已绘制的价值流图中提取端到端全流程 JSON，供全局 ITGap 分析使用。 |
| **全局 ITGap 分析** | 基于 enterprise_context、business_canvas、full_process_vsm，从全局架构失调、决策协同断裂、数字化覆盖盲区、优先级建议矩阵等维度产出结构化分析。 |
| **角色与权限模型推演 (Task10)** | 基于 VSM 节点识别所有业务干系人，定义其在企业微信和低代码系统中的职责边界与访问范围。 |
| **核心业务对象推演 (Task11)** | 定义流程中流转的数字化实体（如订单、合同、任务单），为低代码数据库建模提供底层逻辑结构。 |
| **全局架构设计 (Task12)** | 基于全局 ITGap 分析，设计整体 IT 架构与系统边界，明确各模块职责与集成关系。 |
| **环节专项设计 (Task13)** | 针对各业务环节的局部 ITGap，进行专项方案设计，输出可落地的功能与接口需求。 |
| **链条串联与闭环 (Task14)** | 串联端到端流程各环节方案，形成闭环实施路径与优先级。 |
| **价值回溯与自检 (Task15)** | 将解决方案对标原始 IT Gap 清单，验证有效性并评估预期业务收益。 |

### 3.4 IT 策略规划阶段

| 任务 | 评价标准 |
|------|----------|
| **角色与权限模型推演 (Task10)** | 覆盖 VSM 中所有动作的发出者、接收者及监管者；实现内外协同权限物理隔离；用户确认后视为完成。 |
| **核心业务对象推演 (Task11)** | 对象字段足以支撑 IT Gap 分析需求；建立清晰生命周期状态机；用户确认后视为完成。 |
| **全局架构设计 (Task12)** | 各系统间数据接口定义清晰、标准化；用户可在企微单一工作台完成全链路操作；用户确认后视为完成。 |
| **环节专项设计 (Task13)** | 消息推送含决策关键上下文；操作层级在 3 层以内；用户确认后视为完成。 |
| **链条串联与闭环 (Task14)** | 跨节点信息传递无断点；流程分支有对应数字化逻辑闭环；用户确认后视为完成。 |
| **价值回溯与自检 (Task15)** | IT Gap 清单 100% 功能覆盖；可给出效率提升或质量改善指标；用户确认后视为完成。 |

### 3.5 完成状态存储

- **completedStages**：需求理解子步骤完成标记，`[0, 1, 2]` 分别对应企业背景洞察、商业画布加载、需求逻辑构建。
- **workflowAlignCompletedStages**：工作流对齐子步骤完成标记，`[0, 1, 2]` 分别对应绘制价值流、IT 现状标注、痛点标注。
- **itGapCompletedStages**：ITGap 分析子步骤完成标记，`[0, 1, 2]` 分别对应端到端流程绘制、全局 ITGap 分析、局部 ITGap 分析。
- **globalItGapAnalysisJson**：全局 ITGap 分析的结构化 JSON 结果，确认后写入工作区「全局ITGap 分析」卡片。
- **localItGapSessions**：局部 ITGap 分析 session 列表，每项含 stepName、stepIndex、prompt、analysisJson、analysisMarkdown；确认 session 列表后生成，逐环节分析完成后更新。
- **localItGapAnalyses**：局部 ITGap 分析结果列表，与 localItGapSessions 对应，用于工作区展示。
- **clearDigitalProblemLocalItGapStep**（storage）：支持按环节清除局部 ITGap 某环节的分析结果，用于重做单环节。
- **currentMajorStage**：当前已到达的大阶段索引（0=需求理解，1=工作流对齐，2=ITGap分析，3=IT策略规划）。
- **itStrategyPlanViewingSubstep**：IT 策略规划阶段当前选中的任务索引（0=角色与权限 Task10，1=核心业务对象 Task11，2=全局架构 Task12，3=环节专项 Task13，4=链条串联 Task14，5=价值回溯 Task15），用于工作区任务按钮栏高亮与内容区切换。打开问题详情时若处于 IT 策略规划阶段，根据「第一个未完成任务」恢复本值，使刷新后停留在当前任务页。

## 5. 聊天区的任务提示交互设计

### 5.1 设计原则

- **前置条件驱动**：仅当满足前置任务完成条件时，才展示下一任务的「即将开始」提示块。
- **用户确认**：每个「即将开始」类提示均需用户点击「确认」后才真正发起大模型调用。
- **状态反馈**：确认后按钮变为「已确认」并禁用，避免重复触发。
- **任务完工确认优先**：当出现任务完工确认块（「【任务名】是否视为完成？」）时，**下一任务的开始通知**仅当用户点击「已完成」后才弹出；不得在用户尚未点击「已完成」时提前展示下一任务通知。

### 5.2 需求理解阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 任务通知：我即将开始【企业背景洞察】任务 | 当前第一个未完成任务为 task1，且聊天中尚无该任务启动通知 | 仅在聊天区推送系统消息「请提供企业基本工商信息」，**不**发送任务上下文内容卡片（taskContextBlock）；刷新工作区与标题栏 |
| 我将发起商业模式画布 BMC 生成 | 企业背景洞察已完成（basicInfo 已确认） | 调用 `generateBmcFromBasicInfo`，展示 BMC 卡片供用户确认 |
| 我即将开始提取需求逻辑 | 商业画布加载已完成（bmc 已确认） | 调用 `generateRequirementLogicFromInputs`，展示需求逻辑块供用户确认 |

### 5.3 页面刷新时的自动提示块策略（统一逻辑）

无论是首次进入问题详情还是刷新页面，只要打开问题详情页，系统会根据当前问题记录自动判断「下一个要开始的任务」，并自动尝试发起对应任务的提示块。整体策略如下：

- 初始化问题详情聊天区时（`initProblemDetailChat`）：
  - 总是尝试依次调用：
    - `maybeShowBmcStartBlock()`（Task2：商业画布加载）
    - `maybeShowRequirementLogicStartBlock()`（Task3：需求逻辑构建）
    - `maybeShowValueStreamStartBlock()`（Task4：绘制价值流）
    - `maybeShowItStatusStartBlock()`（Task5：IT 现状标注）
    - `maybeShowPainPointStartBlock()`（Task6：痛点标注）
    - `maybeShowItGapStartBlock()`（若有，为 ITGap 阶段做兜底）
  - 若当前大阶段已到达 ITGap 分析（`currentMajorStage >= 2`），则额外通过 `requestAnimationFrame` 调用：
    - `maybeShowE2eFlowExtractBlock()`（Task7：端到端流程绘制）
    - `maybeShowGlobalItGapStartBlock()`（Task8：全局 ITGap 分析）
    - `forceShowLocalItGapStartBlock()`（Task9：局部 ITGap 分析）
- 渲染 IT 策略规划工作区时（`renderProblemDetailContent` 且 `problemDetailViewingMajorStage === 3`）：
  - 始终渲染顶部任务按钮栏（Task10–Task15），并根据 `itStrategyPlanViewingSubstep` 决定当前选中任务；
  - 当当前子任务为 Task10（角色与权限模型推演）时，通过 `requestAnimationFrame` 调用 `maybeShowRolePermissionStartBlock()`：
    - 若尚未出现 `rolePermissionStartBlock` 且 `currentMajorStage === 3`，则在聊天区自动展示「即将开始角色与权限模型推演」提示块。

> **注意**：每个 `maybeShowXXXStartBlock` 内部都会检查前置条件（如：前一任务是否已完成、必要输入是否就绪、是否已经存在同类 startBlock、当前大阶段是否已超前等），因此多次刷新或重复调用不会导致重复提示。

**刷新后子步骤与滚动恢复**：打开问题详情时（含从路由恢复），若当前大阶段为 IT 策略规划（`currentMajorStage === 3`），根据「第一个未完成任务」设置 `itStrategyPlanViewingSubstep`（task10→0, task11→1, …），使刷新后停留在当前任务页而非总是回到 Task10。工作区滚动位置：在 `pagehide` 时若当前在问题详情且存在 `.problem-detail-workspace-scroll`，将其 `scrollTop` 存入 sessionStorage（key: `problemDetailScroll_${item.createdAt}`）；再次打开同一问题详情并执行 `renderProblemDetailContent` 后，在双 rAF 内读取该 key、恢复滚动位置并删除该 key。

### 5.4 工作流对齐阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 我即将开始需求相关核心价值流图绘制 | currentMajorStage≥1，且 workflowAlignCompletedStages 不包含 0，且 basicInfo/bmc/requirementLogic 就绪 | 调用 `generateValueStreamFromInputs`，展示价值流 JSON 卡片 |
| 开始绘制价值流图 | 价值流 JSON 已生成且用户已确认价值流卡片 | 将 valueStream 写入问题记录，workflowAlignCompletedStages 写入 0，刷新工作区 |
| 即将开始 IT 现状标注 | workflowAlignCompletedStages 包含 0 且不包含 1，requirementLogic 就绪 | 调用 `generateItStatusAnnotation`，为各环节标注 itStatus |
| 即将开始价值流图环节节点痛点标注 | workflowAlignCompletedStages 包含 0、1 且不包含 2，requirementLogic 就绪 | 调用 `generatePainPointAnnotation`，为各环节标注 painPoint |

### 5.5 ITGap 分析阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 我先需要提取端到端流程绘制的 json 数据 | itGapCurrent === 0（端到端流程绘制未完成）、且存在 valueStream | 在聊天区展示端到端流程 JSON 卡片（四按钮：确认、重做、修正、讨论）；**推送时**即在沟通历史过程日志中增加一条标签为「输出」的时间线（含该 JSON 数据）。用户点击「确认」：该条变为「确认」，聊天区推送「【端到端流程绘制】是否视为完成？」；用户点击「已完成」：过程日志增加「任务完成」（内容「用户确认端到端流程绘制」），当前任务变为 task8。点击「重做」可将该块置为未确认、时间线恢复为「输出」。 |

**全局 ITGap 分析结构化卡片维度**：全局架构失调诊断 (Structural Gap)、决策协同断裂分析 (Collaboration Gap)、数字化覆盖盲区 (Digital Blind Spots)、优先级建议矩阵 (Roadmap Strategy)、全局洞察结论、As-Is（现状）与 To-Be（目标）、Top3 核心 IT 缺口。

**工作区展示**：确认后在工作区端到端流程卡片下方新增「全局ITGap 分析」卡片，标题栏含 Tab「全局ITGap 分析」/「JSON」可切换查看。

### 5.7 对话模式（Agent / Ask）

问题详情对话区支持两种模式，通过输入框左上角的下拉选择器切换（Cursor 风格，图标 + 文本 +  chevron）：

| 模式 | 说明 |
|------|------|
| **Agent** | 与大模型完整联动：意图提炼后执行修改/执行任务会更新工作区；讨论、修改等会写入沟通历史；对话标题栏显示**「当前任务：【当前任务名称】」**，任务名以蓝底白字标签突出显示；**点击该标签**可在工作区自动切换到对应大阶段并滚动到对应内容卡片（需求理解/工作流对齐/ITGap/IT策略规划及子步骤卡片）；输入框 placeholder 为「输入消息或执行操作…」 |
| **Ask** | 仅支持查询与讨论：用户输入只触发查询/讨论，不更新工作区，不写入沟通历史；意图为「修改」「执行」时展示提示「当前为 Ask 模式，仅支持查询与讨论。请切换至 Agent 模式」；对话标题栏显示**「询问讨论模式」**；**输入框**：底色为淡绿色（#e8f5e9）、字体为深蓝色（#0d47a1）、占位符为「您想了解本项目的什么信息？」，以与背景色形成反差、便于识别 |

- **消息标记**：Ask 模式下产生的消息带 `askMode: true`，`js/communication-history.js` 的 `getCommunicationsByTask` 会排除这些消息，故不出现在沟通历史中。
- **发送按钮**：大模型处理期间（意图提炼、查询、讨论、修改等）发送按钮禁用，处理完成后恢复。
- **上下文截断提示**：Ask 模式查询时，`buildAskModeProblemContext` 将各数据块限制在 `ASK_CONTEXT_MAX_PER_BLOCK`（20000 字符）内；若任一数据块被截断，则在 LLM 回答展示后，聊天区追加系统提示「上下文长度已经超过最大截断值，查询信息不一定准确。」

### 5.8 确认 / 修改 / 讨论 与任务完工确认流程

对所有大模型返回的提炼内容块（如企业基本信息、BMC、需求逻辑、价值流、意图提炼、角色与权限、全局/局部 ITGap、修改/讨论后的新提炼块等），在「确认」按钮旁提供**「修改」**与**「讨论」**按钮（样式与「重做」按钮一致：灰字、透明底、描边）。

- **修改**：用户点击「修改」后，（1）**不**向当前任务的过程日志推送任何内容；（2）在聊天区追加系统提示「请描述您的修改意见（输入后发送）」；（3）用户输入并发送后，系统以**当前任务的时间线历史**为上下文 + 用户反馈，调用大模型重新生成；（4）大模型回复作为新的提炼内容块（`modificationResponseBlock`）展示，同样带确认/修改/讨论按钮。用户发送的修改意见以「修正」标签写入过程日志。
- **讨论**：用户点击「讨论」后，在聊天区追加系统提示「请描述您想讨论的问题（输入后发送）」；用户输入并发送后，同样以当前任务时间线 + 用户输入调用大模型，回复作为新的内容块展示。

用户点击「确认」后触发统一流程：

1. **过程日志更新**：按该内容块的时间戳及「确认」标签，在当前任务的过程日志时间线中更新/展示对应条目。
2. **任务完工确认块**：在聊天区发送任务完工确认内容块：「【当前任务名称】是否视为完成？」，下方提供两个按钮：
   - **已完成**：（1）在当前任务的过程日志时间线中增加一条「任务完成」类型的内容块（**仅写入过程日志，不在聊天区展示**），展示内容按任务区分（如端到端流程绘制为「用户确认端到端流程绘制」、全局 ITGap 分析为「用户确认全局 ITGap 分析」等），标签为「任务完成」；（2）将当前问题的**当前状态推进到状态机下一状态**（如标记本任务完成、更新 `completedStages` / `workflowAlignCompletedStages` / `itGapCompletedStages` / `currentMajorStage` 或 `completedTaskIds`，并刷新工作区与进度）；（3）**弹出下一任务的开始通知**；（4）确认块上按钮置为已选/禁用。
   - **还不行**：在聊天框中追加系统消息「请您进一步反馈修改意见」；确认块上按钮置为已选/禁用。「还不行」按钮的视觉样式与「重做」按钮一致（灰字、透明底、描边）。

上述流程在问题详情聊天区所有带「确认」的提炼内容块上统一生效（任务启动类「确认」除外）。**端到端流程 JSON 卡片**与其它提炼块一致：点击「确认」后仅更新过程日志中该条为「确认」并推送「是否视为完成」块，**不**在此刻推进任务状态；仅当用户点击「已完成」后才写入「任务完成」条目并推进当前任务至全局 ITGap 分析。

## 6. 工作区与阶段导航

### 6.0 工作区内容卡片 UI 规范

- **子标题与正文字号**：工作区中内容卡片（如客户名称、客户需求或挑战、客户IT现状、项目时间要求等 key-value 行）的**正文（.problem-detail-value）字号不得大于子标题（.problem-detail-label）字号**。实现上子标题采用固定字号（如 0.9375rem），正文采用相同或更小字号（如 0.875rem），以保证层级清晰。

### 6.1 阶段导航条

- 展示四个大阶段：需求理解 → 工作流对齐 → ITGap分析 → IT策略规划
- 已完成的阶段显示 ✅，可点击回看
- 未到达的阶段不可点击
- 「回退」按钮位于「重置」右侧，橙色底白字；点击后问题单回到上一任务阶段、清空上一阶段全部数据，并触发上一阶段的任务开始通知；已在阶段 0 时无法回退

### 6.2 子步骤展示

- **需求理解**：企业背景洞察 → 商业画布加载 → 需求逻辑构建
- **工作流对齐**：绘制价值流 → IT 现状标注 → 痛点标注
- **ITGap 分析**：端到端流程绘制 → 全局 ITGap 分析 → 局部 ITGap 分析
- **IT 策略规划**：角色与权限 (Task10) → 核心业务对象 (Task11) → 全局架构 (Task12) → 环节专项 (Task13) → 链条串联 (Task14) → 价值回溯 (Task15)

子步骤以横向链条形式展示，已完成显示 ✅，当前进行中高亮。IT 策略规划的六个任务（Task10–Task15）为可点击切换的按钮栏，无完成状态标记。

- **需求理解**与 **ITGap 分析** 的子步骤支持点击：点击后自动滚动到对应工作区卡片并展开（如点击「企业背景洞察」→ 客户基本信息卡片，「端到端流程绘制」→ 端到端全流程卡片）。**ITGap 工作区「端到端流程」卡片**：在用户于聊天区对端到端流程 JSON 点击「已完成」之前显示占位提示（如「请先在聊天区确认端到端流程 JSON 后，此处将显示端到端流程」）；用户点击「已完成」后，工作区该卡片显示端到端流程内容。
