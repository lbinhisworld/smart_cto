# 数字化问题跟进阶段设计

本文档定义数字化问题从创建到解决的全过程任务、阶段划分、任务目标与评价体系、阶段切换触发事件、**状态管理模块**，以及聊天区的任务提示交互设计。

---

## 0. 状态管理模块（每个数字化问题单）

### 0.1 15 种任务状态（即任务名称）

每个数字化问题单均包含以下 15 种状态，每种状态对应一个任务名称，**依次顺序触发**：

| 序号 | 任务 ID | 任务名称（状态名） |
|------|---------|--------------------|
| ① | task1 | 企业背景洞察 |
| ② | task2 | 商业画布加载 |
| ③ | task3 | 需求逻辑构建 |
| ④ | task4 | 绘制价值流 |
| ⑤ | task5 | IT现状标注 |
| ⑥ | task6 | 痛点标注 |
| ⑦ | task7 | 端到端流程绘制 |
| ⑧ | task8 | 全局 ITGap 分析 |
| ⑨ | task9 | 局部 ITGap 分析 |
| ⑩ | task10 | 角色与权限模型推演 |
| ⑪ | task11 | 核心业务对象推演 |
| ⑫ | task12 | 全局架构设计 |
| ⑬ | task13 | 环节专项设计 |
| ⑭ | task14 | 链条串联与闭环 |
| ⑮ | task15 | 价值回溯与自检 |

### 0.2 顺序触发与当前任务追踪

- **顺序触发**：状态①结束后，当前任务切换到状态②；状态②结束后，当前任务切换到状态③；依次类推。
- **当前任务**：系统对每个问题单维护一个「当前任务状态」，表示当前应执行或正在执行的任务。当前任务 = 下一个待完成的状态（第一个未完成的 task 即为当前任务）。
- **状态推进**：任一任务完成（满足评价标准并用户确认）后，当前任务自动指向下一个未完成任务；若所有 15 个任务均已完成，则当前任务为「已全部完成」。

### 0.3 任务扩展字段（在原有任务字段基础上）

每个任务（或每个问题单下针对每个任务的状态记录）增加如下字段：

| 字段 | 说明 |
|------|------|
| **输入** | 该任务所需的前置输入（如：企业基本信息、BMC、需求逻辑、价值流图等），用于描述本任务的触发前提 |
| **动作** | 该任务的核心动作描述（如：提炼企业信息、生成 BMC、绘制价值流、标注 IT 现状等） |
| **输出反馈** | 该任务完成后的输出物及用户反馈形式（如：结构化 JSON、Markdown 卡片、用户确认等） |
| **任务状态** | 本任务在此问题单下的状态：`pending`（待执行）、`in_progress`（进行中）、`completed`（已完成） |

### 0.4 各任务的输入 / 动作 / 输出反馈 / 任务状态示例

| 任务 | 输入 | 动作 | 输出反馈 | 任务状态 |
|------|------|------|----------|----------|
| ① 企业背景洞察 | 客户输入的企业信息 | 提炼并结构化企业基本信息 | 包含核心字段的 JSON，用户确认 | pending / in_progress / completed |
| ② 商业画布加载 | 企业基本信息 | 运用 BMC 框架生成商业分析 | BMC JSON，用户确认 | pending / in_progress / completed |
| ③ 需求逻辑构建 | 基本信息 + BMC + 初步需求 | 产出需求逻辑链条分析 | 需求逻辑 Markdown，用户确认 | pending / in_progress / completed |
| ④ 绘制价值流 | enterprise_info + bmc_data + requirement_logic | 生成价值流 JSON 并绘制 | 价值流 JSON + 用户点击「开始绘制」 | pending / in_progress / completed |
| ⑤ IT现状标注 | 价值流图 + 需求逻辑 | 在各环节标注 IT 支撑方式 | 各环节 itStatus，用户确认 | pending / in_progress / completed |
| ⑥ 痛点标注 | 价值流图 + 需求逻辑 | 在各环节提炼痛点 | 各环节 painPoint，用户确认 | pending / in_progress / completed |
| ⑦ 端到端流程绘制 | 已绘制价值流图 | 提取端到端全流程 JSON | 端到端流程 JSON，用户确认 | pending / in_progress / completed |
| ⑧ 全局 ITGap 分析 | 企业上下文 + BMC + 全流程 | 从多维度产出全局 ITGap 分析 | 结构化 JSON 卡片，用户确认 | pending / in_progress / completed |
| ⑨ 局部 ITGap 分析 | 全局 ITGap + 端到端流程 | 逐环节生成局部 ITGap 分析 | 各环节 analysis，写入 Task9 日志 | pending / in_progress / completed |
| ⑩ 角色与权限模型推演 | 价值流图（环节与角色） | 识别干系人并定义权限边界 | Markdown 表格，用户确认 | pending / in_progress / completed |
| ⑪ 核心业务对象推演 | 流程与 IT Gap | 定义数字化实体与状态机 | 业务对象 JSON，用户确认 | pending / in_progress / completed |
| ⑫ 全局架构设计 | 全局 ITGap 分析 | 设计 IT 架构与系统边界 | 架构图/文档，用户确认 | pending / in_progress / completed |
| ⑬ 环节专项设计 | 局部 ITGap | 针对各环节设计专项方案 | 功能/接口需求，用户确认 | pending / in_progress / completed |
| ⑭ 链条串联与闭环 | 环节方案 | 串联端到端，形成闭环 | 实施路径与优先级，用户确认 | pending / in_progress / completed |
| ⑮ 价值回溯与自检 | IT Gap 清单 + 方案 | 对标验证并评估收益 | 对冲率与价值量化，用户确认 | pending / in_progress / completed |

### 0.5 与既有存储的映射

- **currentTaskIndex / currentTaskId**：可由既有 `completedStages`、`workflowAlignCompletedStages`、`itGapCompletedStages`、`currentMajorStage` 等推导出「当前任务」为 task1–task15 中的哪一个。
- **任务状态（task status）**：已完成任务对应 `completed`；当前正在执行的任务对应 `in_progress`；尚未到达的任务对应 `pending`。
- **输入 / 动作 / 输出反馈**：可从现有任务定义（objective、evaluationCriteria）及触发事件中提炼，用于任务卡片或工作区展示。

### 0.6 沟通历史中任务扩展字段的展示（已实现）

在问题详情页「沟通历史」侧滑面板中，每个任务为可折叠卡片。展开后包含**两个 Tab**：「任务详情」与「过程日志」。

- **任务详情 Tab**：展示归属阶段、任务目标、评估标准、**输入**（该任务的前置输入说明）、**动作**（核心动作描述）、**输出反馈**（完成后的输出物及用户反馈形式）、**任务状态**（由当前问题单推导：`已完成` / `进行中` / `待执行`）。任务过程日志仅从既有聊天记录提取，不再写入新条目（相关 Log 类写入已移除）。
- **过程日志 Tab**：以时间线展示该任务下的沟通记录。每条时间线内容块的**标题栏由时间戳 + 日志类型标签**构成；日志类型为：**输入**（用户发言）、**确认**（执行/查询/系统确认等）、**修改**（意图为修改）、**讨论**（意图为讨论或系统提炼）。详见《对话模型管理》4.5。

**任务标题栏显示**：已完成任务的标题栏字体为**绿色**；进行中为**蓝色**；待执行沿用默认文本色。CSS 类：`problem-detail-history-task-done`、`problem-detail-history-task-current`。

数据来源：`TASK_EXTRA_FIELDS`（js/config.js）为每个 taskId 提供 input、action、outputFeedback；任务状态由 `isTaskCompleted` 及「第一个未完成任务」逻辑计算得出。

## 2. 任务目标设计

| 任务 | 目标描述 |
|------|----------|
| **企业背景洞察** | 基于客户输入的企业信息，提炼并结构化企业基本信息，为后续 BMC 与需求逻辑分析提供基础。 |
| **商业画布加载** | 基于企业基本信息，运用商业模式画布（BMC）框架生成结构化商业分析，识别客户细分、价值主张、渠道通路等九大模块。 |
| **需求逻辑构建** | 基于客户初步需求、企业基本信息、BMC 三个维度，产出需求背后的逻辑链条分析，明确行业特性、商业模式局限与业务痛点的因果关联。 |
| **绘制价值流** | 基于 enterprise_info、bmc_data、requirement_logic 生成业务核心价值流图，划分阶段与环节，标注执行角色与预估耗时。 |
| **IT 现状标注** | 结合需求逻辑，在价值流图每个环节节点标注该环节的 IT 支撑方式（手工/系统），区分纸质、excel 或具体系统名称。 |
| **痛点标注** | 结合需求逻辑，在价值流图每个环节节点提炼该环节涉及到的痛点，为后续 IT Gap 分析提供输入。 |
| **端到端流程绘制** | 从已绘制的价值流图中提取端到端全流程 JSON，供全局 ITGap 分析使用。 |
| **全局 ITGap 分析** | 基于 enterprise_context、business_canvas、full_process_vsm，从全局架构失调、决策协同断裂、数字化覆盖盲区、优先级建议矩阵等维度产出结构化分析。 |
| **全局架构设计 (Task10)** | 基于全局 ITGap 分析，设计整体 IT 架构与系统边界，明确各模块职责与集成关系。 |
| **环节专项设计 (Task11)** | 针对各业务环节的局部 ITGap，进行专项方案设计，输出可落地的功能与接口需求。 |
| **链条串联与闭环 (Task12)** | 串联端到端流程各环节方案，形成闭环实施路径与优先级，输出 IT 策略规划结论。 |

## 3. 任务评价体系设计

### 3.1 需求理解阶段

| 任务 | 评价标准 |
|------|----------|
| **企业背景洞察** | 成功提取并确认包含公司名称、信用代码、法人、成立时间、注册资本、经营范围等核心字段的结构化 JSON。 |
| **商业画布加载** | 生成完整的 BMC JSON，包含行业背景洞察、九大画布模块及业务痛点预判，用户确认后视为完成。 |
| **需求逻辑构建** | 输出结构化的需求逻辑 Markdown，包含行业底层逻辑、因果关联、深层动机及逻辑链条总结，用户确认后视为完成。 |

### 3.2 工作流对齐阶段

| 任务 | 评价标准 |
|------|----------|
| **绘制价值流** | 生成符合前端绘图要求的价值流 JSON，用户确认并完成「开始绘制价值流图」后视为完成。 |
| **IT 现状标注** | 每个环节均标注 itStatus，用户确认后视为完成。 |
| **痛点标注** | 每个环节均标注 painPoint（无明显痛点的环节不展示卡片），用户确认后视为完成。 |

### 3.3 ITGap 分析阶段

| 任务 | 评价标准 |
|------|----------|
| **端到端流程绘制** | 用户确认端到端流程 JSON 后视为完成；任务过程日志不再写入新条目，仅从聊天记录提取展示。 |
| **全局 ITGap 分析** | 大模型生成分析 → 用户确认 JSON 形成结构化卡片 → 用户确认结构化卡片后写入工作区，视为完成；任务过程日志不再写入新条目。 |
| **局部 ITGap 分析** | 用户确认「即将生成每个环节的 ITGap 分析 session」→ 系统生成 session 列表 → 用户点击「确认」或「继续」→ 系统逐环节调用大模型分析，每完成一环更新工作区与聊天区，全部完成视为完成；任务过程日志不再写入新条目。 |

### 3.4 IT 策略规划阶段

| 任务 | 评价标准 |
|------|----------|
| **全局架构设计 (Task10)** | 内容展示区可切换查看，后续接入大模型生成与编辑后由用户确认视为完成。 |
| **环节专项设计 (Task11)** | 同上。 |
| **链条串联与闭环 (Task12)** | 同上。 |

### 3.5 完成状态存储

- **completedStages**：需求理解子步骤完成标记，`[0, 1, 2]` 分别对应企业背景洞察、商业画布加载、需求逻辑构建。
- **workflowAlignCompletedStages**：工作流对齐子步骤完成标记，`[0, 1, 2]` 分别对应绘制价值流、IT 现状标注、痛点标注。
- **itGapCompletedStages**：ITGap 分析子步骤完成标记，`[0, 1, 2]` 分别对应端到端流程绘制、全局 ITGap 分析、局部 ITGap 分析。
- **globalItGapAnalysisJson**：全局 ITGap 分析的结构化 JSON 结果，确认后写入工作区「全局ITGap 分析」卡片。
- **localItGapSessions**：局部 ITGap 分析 session 列表，每项含 stepName、stepIndex、prompt、analysisJson、analysisMarkdown；确认 session 列表后生成，逐环节分析完成后更新。
- **localItGapAnalyses**：局部 ITGap 分析结果列表，与 localItGapSessions 对应，用于工作区展示。
- **currentMajorStage**：当前已到达的大阶段索引（0=需求理解，1=工作流对齐，2=ITGap分析，3=IT策略规划）。
- **itStrategyPlanViewingSubstep**：IT 策略规划阶段当前选中的任务索引（0=角色与权限 Task10，1=核心业务对象 Task11，2=全局架构 Task12，3=环节专项 Task13，4=链条串联 Task14，5=价值回溯 Task15），用于工作区任务按钮栏高亮与内容区切换。

## 4. 任务阶段切换的触发事件

### 4.1 需求理解 → 工作流对齐

**触发条件**：用户在工作流对齐占位页中，于聊天区确认价值流图设计 JSON 后，点击「开始绘制价值流图」卡片的「确认」按钮。

- `currentMajorStage` 更新为 1

- 聊天区自动检测并展示「即将开始 IT 现状标注」或「即将开始痛点标注」等后续提示块（若满足前置条件）

|-------|----------|
| 绘制价值流 | 用户点击「我即将开始需求相关核心价值流图绘制」→ 确认 → 大模型生成价值流 JSON → 用户点击价值流卡片的「确认」→ 用户点击「开始绘制价值流图」的「确认」 |

### 4.4 ITGap 分析 → IT 策略规划

**触发条件**：局部 ITGap 分析全部完成（所有环节均已生成分析结果）。

**触发动作**：

用户通过聊天区发送「重新进行 XXX」类指令，经意图提炼识别为 `intent=execute` 时，可直接触发对应任务：

## 5. 聊天区的任务提示交互设计

### 5.1 设计原则

- **前置条件驱动**：仅当满足前置任务完成条件时，才展示下一任务的「即将开始」提示块。
- **用户确认**：每个「即将开始」类提示均需用户点击「确认」后才真正发起大模型调用。
- **状态反馈**：确认后按钮变为「已确认」并禁用，避免重复触发。

### 5.2 需求理解阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 我将发起商业模式画布 BMC 生成 | 企业背景洞察已完成（basicInfo 已确认） | 调用 `generateBmcFromBasicInfo`，展示 BMC 卡片供用户确认 |
| 我即将开始提取需求逻辑 | 商业画布加载已完成（bmc 已确认） | 调用 `generateRequirementLogicFromInputs`，展示需求逻辑块供用户确认 |

### 5.3 页面刷新时的自动提示块策略（统一逻辑）

无论是首次进入问题详情还是刷新页面，只要打开问题详情页，系统会根据当前问题记录自动判断「下一个要开始的任务」，并自动尝试发起对应任务的提示块。整体策略如下：

- 初始化问题详情聊天区时（`initProblemDetailChat`）：
  - 总是尝试依次调用：
    - `maybeShowBmcStartBlock()`（Task2：商业画布加载）
    - `maybeShowRequirementLogicStartBlock()`（Task3：需求逻辑构建）
    - `maybeShowValueStreamStartBlock()`（Task4：绘制价值流）
    - `maybeShowItStatusStartBlock()`（Task5：IT 现状标注）
    - `maybeShowPainPointStartBlock()`（Task6：痛点标注）
    - `maybeShowItGapStartBlock()`（若有，为 ITGap 阶段做兜底）
  - 若当前大阶段已到达 ITGap 分析（`currentMajorStage >= 2`），则额外通过 `requestAnimationFrame` 调用：
    - `maybeShowE2eFlowExtractBlock()`（Task7：端到端流程绘制）
    - `maybeShowGlobalItGapStartBlock()`（Task8：全局 ITGap 分析）
    - `forceShowLocalItGapStartBlock()`（Task9：局部 ITGap 分析）
- 渲染 IT 策略规划工作区时（`renderProblemDetailContent` 且 `problemDetailViewingMajorStage === 3`）：
  - 始终渲染顶部任务按钮栏（Task10–Task15），并根据 `itStrategyPlanViewingSubstep` 决定当前选中任务；
  - 当当前子任务为 Task10（角色与权限模型推演）时，通过 `requestAnimationFrame` 调用 `maybeShowRolePermissionStartBlock()`：
    - 若尚未出现 `rolePermissionStartBlock` 且 `currentMajorStage === 3`，则在聊天区自动展示「即将开始角色与权限模型推演」提示块。

> **注意**：每个 `maybeShowXXXStartBlock` 内部都会检查前置条件（如：前一任务是否已完成、必要输入是否就绪、是否已经存在同类 startBlock、当前大阶段是否已超前等），因此多次刷新或重复调用不会导致重复提示。

### 5.4 工作流对齐阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 我即将开始需求相关核心价值流图绘制 | currentMajorStage≥1，且 workflowAlignCompletedStages 不包含 0，且 basicInfo/bmc/requirementLogic 就绪 | 调用 `generateValueStreamFromInputs`，展示价值流 JSON 卡片 |
| 开始绘制价值流图 | 价值流 JSON 已生成且用户已确认价值流卡片 | 将 valueStream 写入问题记录，workflowAlignCompletedStages 写入 0，刷新工作区 |
| 即将开始 IT 现状标注 | workflowAlignCompletedStages 包含 0 且不包含 1，requirementLogic 就绪 | 调用 `generateItStatusAnnotation`，为各环节标注 itStatus |
| 即将开始价值流图环节节点痛点标注 | workflowAlignCompletedStages 包含 0、1 且不包含 2，requirementLogic 就绪 | 调用 `generatePainPointAnnotation`，为各环节标注 painPoint |

### 5.5 ITGap 分析阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 我先需要提取端到端流程绘制的 json 数据 | itGapCurrent === 1 且 Task7 沟通历史为空 | 展示端到端流程 JSON 卡片，用户确认后视为完成（过程日志不再写入） |
| 即将针对端到端流程开展全局 ITGap 分析 | itGapCurrent === 1、Task7 沟通历史非空、无 globalItGapAnalysisJson | 调用 `runGlobalItGapAnalysis`，展示 JSON 卡片；用户确认 JSON 后形成结构化卡片；用户确认结构化卡片后写入工作区，视为完成（过程日志不再写入） |
| 即将生成每个环节的 ITGap 分析 session | itGapCurrent === 2、有 globalItGapAnalysisJson、端到端流程有环节 | 用户确认后生成 session 列表（不调用大模型）；用户点击「确认」或「继续」后逐环节调用大模型分析，每完成一环更新工作区、聊天区（过程日志不再写入） |
| 即将开始 IT 策略规划 | 局部 ITGap 分析全部完成（所有环节已分析） | 用户确认后，`currentMajorStage` 更新为 3，跳转到 IT 策略规划页面 |

**IT 策略规划页面结构**（与 ITGap 分析一致）：左侧对话区，右侧工作区。工作区顶部为任务按钮栏（样式与 ITGap 分析子步骤栏一致），三个任务按钮：**全局架构设计 (Task10)**、**环节专项设计 (Task11)**、**链条串联与闭环 (Task12)**；点击切换当前任务并刷新下方内容展示区。**Task10 角色与权限模型推演**：当存在价值流数据时，内容区展示「大阶段可折叠卡片 + 环节可折叠子卡片」结构；一级卡片左边缘与按钮栏左边缘对齐，环节子卡片嵌入大阶段内并带树形引导线，推演结果确认后填充各环节详情，未推演环节显示「待推演」。

**全局 ITGap 分析结构化卡片维度**：全局架构失调诊断 (Structural Gap)、决策协同断裂分析 (Collaboration Gap)、数字化覆盖盲区 (Digital Blind Spots)、优先级建议矩阵 (Roadmap Strategy)、全局洞察结论、As-Is（现状）与 To-Be（目标）、Top3 核心 IT 缺口。

**工作区展示**：确认后在工作区端到端流程卡片下方新增「全局ITGap 分析」卡片，标题栏含 Tab「全局ITGap 分析」/「JSON」可切换查看。

### 5.6 IT 策略规划阶段提示块

| 提示内容 | 前置条件 | 确认后动作 |
|----------|----------|------------|
| 即将开始 IT 策略规划 | 局部 ITGap 分析全部完成（所有环节已分析），且尚未进入 IT 策略规划 | 用户点击「确认」后，`currentMajorStage` 更新为 3，跳转到 IT 策略规划页面，默认选中 Task10「角色与权限模型推演」；随后触发角色与权限模型推演的开始提示。 |
| 即将开始角色与权限模型推演 | 当前已处于 IT 策略规划阶段（`currentMajorStage === 3`），当前子任务为 Task10，且尚未存在 `rolePermissionStartBlock` | 用户点击「确认」后：（1）在聊天区发送「正在准备工作区」；（2）在工作区绘制基本页面结构：将所有价值流大阶段绘制为可折叠大卡片，每个阶段卡片下为对应环节的可折叠子卡片（子卡片前有树形引导线，阶段卡片 body 上下留白）；（3）完成上述后再调用 `runRolePermissionModeling(false)` 开始大模型推演，生成结构化 Markdown 表格卡片并写入沟通历史与工作区。 |

- **角色与权限确认内容卡片**：聊天区中角色与权限模型推演结果卡片支持删除（带删除按钮）；点击删除后，该条消息从沟通历史移除，工作区根据「最新已确认」结果重绘，若删除的为当前展示用卡片则工作区对应内容同步清除。
- **工作区 Task10 结构**：一级卡片（大阶段）左边缘与任务按钮栏「角色与权限模型推演」按钮左边缘对齐（通过 `.problem-detail-role-permission-wrap` 与 `.problem-detail-substeps` 一致的左右 padding）；环节子卡片以嵌入形式排在大阶段卡片内，左侧有竖线 + 每条环节前横枝的树形引导线；阶段卡片 body 顶部、底部与环节子卡片之间保留空隙。

> 后续 IT 策略规划阶段的其他任务（如核心业务对象推演、全局架构设计、环节专项设计、链条串联与闭环、价值回溯与自检）可沿用同一模式：在满足前置条件时自动展示「即将开始 XXX」提示块，由用户确认后启动对应大模型调用与工作区展示。

### 5.7 对话模式（Agent / Ask）

问题详情对话区支持两种模式，通过输入框左上角的下拉选择器切换（Cursor 风格，图标 + 文本 +  chevron）：

| 模式 | 说明 |
|------|------|
| **Agent** | 与大模型完整联动：意图提炼后执行修改/执行任务会更新工作区；讨论、修改等会写入沟通历史；对话标题栏显示**「当前任务：【当前任务名称】」**，任务名以蓝底白字标签突出显示；**点击该标签**可在工作区自动切换到对应大阶段并滚动到对应内容卡片（需求理解/工作流对齐/ITGap/IT策略规划及子步骤卡片） |
| **Ask** | 仅支持查询与讨论：用户输入只触发查询/讨论，不更新工作区，不写入沟通历史；意图为「修改」「执行」时展示提示「当前为 Ask 模式，仅支持查询与讨论。请切换至 Agent 模式」；对话标题栏显示**「询问讨论模式」** |

- **消息标记**：Ask 模式下产生的消息带 `askMode: true`，`js/communication-history.js` 的 `getCommunicationsByTask` 会排除这些消息，故不出现在沟通历史中。
- **发送按钮**：大模型处理期间（意图提炼、查询、讨论、修改等）发送按钮禁用，处理完成后恢复。
