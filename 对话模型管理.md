# 对话模型管理文档

本文档整理问题详情页聊天区中**用户意图提炼**相关的意图类型定义、解析提示词、过程日志的提取与显示逻辑，便于维护与版本管理。

---

## 1. 用户意图类型

系统支持四种用户意图类型，代码与展示名称的对应关系如下：

| 意图代码 (intent) | 展示名称 | 说明 |
|-------------------|----------|------|
| `query` | 简单查询 | 用户查询当前问题中的具体内容 |
| `modification` | 反馈修改意见 | 用户提出对页面内容的修改需求 |
| `execute` | 执行操作 | 用户执行某任务（如重新构建需求逻辑等） |
| `discussion` | 讨论请教 | 用户针对当前问题进行延展性讨论或请教 |

### 代码中的映射

```javascript
const intentLabels = {
  query: '简单查询',
  modification: '反馈修改意见',
  execute: '执行操作',
  discussion: '讨论请教'
};
```

## 2. 解析提示词

**触发入口**：问题详情页聊天区，用户输入消息后点击发送  
**函数**：`extractUserIntentFromChat(text, context, options)`  
**用途**：提炼用户输入的意图类型，结合沟通历史与页面内容结构定位到具体任务与字段。

**当前任务与「不对」按钮**：
- 默认：系统根据沟通历史推断「当前任务」，并在 prompt 中传入 `currentTaskHint`，大模型优先考虑该任务与用户意图的关联。
- 意图卡片除「确认」外，还有「不对」按钮。用户点击「不对」时，系统重新调用 `extractUserIntentFromChat` 并传入 `globalScope: true`，大模型在【任务列表】中全局搜索，不受当前任务预设限制。

### System Prompt

```
你是一个数字化问题跟进对话的意图分析助手。用户会在聊天区输入消息，请结合【沟通历史】和【当前页面内容结构】提炼意图，从上下文中搜索最为匹配的内容单元，协助定位到对应的页面位置。

【任务列表】
${tasksDesc}

【输出格式】
{
  "taskId": "task1",
  "taskName": "企业背景洞察",
  "stage": "需求理解",
  "intent": "query" | "modification" | "execute" | "discussion",
  "queryTarget": "用户想查询的具体内容，仅当 intent 为 query 时填写",
  "discussionTopic": "用户想请教或讨论的具体话题，仅当 intent 为 discussion 时填写",
  "queryValueStreamLevel": "step" | "stage" | "card",
  "queryValueStreamTarget": "环节名或阶段名",
  "modificationTarget": "用户想修改的具体内容或目标，仅当 intent 为 modification 时填写",
  "modificationField": "具体要修改的字段名称，仅当 intent 为 modification 且能明确到具体字段时填写",
  "modificationValueStreamLevel": "step" | "stage" | "card",
  "modificationValueStreamTarget": "环节名或阶段名",
  "modificationClear": true | false,
  "modificationNewValue": "用户希望修改成的具体内容",
  "executeTaskId": "task2",
  "executeTaskName": "商业画布加载",
  "summary": "一句话概括用户意图"
}

【可修改字段参考】（modificationField 应使用以下精确字段名之一）
- 初步需求：客户名称、客户需求或挑战、客户IT现状、项目时间要求
- 客户基本信息：公司名称、信用代码、法人、成立时间、注册资本、是否上市、上市地、经营范围、核心资质、官方网站
- BMC：行业背景洞察、客户细分、价值主张、渠道通路、客户关系、收入来源、核心资源、关键业务、重要合作、成本结构、业务痛点预判
- 需求逻辑：行业底层逻辑与竞争共性、初步需求与商业模式的"因果关联"、需求背后的深层动机、逻辑链条总结

规则：
1. 结合【沟通历史】理解对话脉络，从【当前页面内容结构】中搜索与用户输入最为匹配的内容单元（字段名、环节名、阶段名等）。
2. taskId/taskName/stage：根据用户消息及沟通历史推断当前沟通涉及的任务及阶段，从上述任务列表中选择最相关的。
3. intent：简单查询(query) / 反馈修改意见(modification) / 执行操作(execute) / 请教讨论(discussion)
4. 若 intent=discussion：用户针对当前问题的各种专题进行延展性讨论或请教时填写。判断用户讨论话题与哪个任务最为相关，填写 taskId；填写 discussionTopic 概括讨论话题。
5. 若 intent=query，填写 queryTarget；若涉及价值流图，从【当前页面内容结构】中匹配环节名/阶段名，填写 queryValueStreamLevel 和 queryValueStreamTarget
6. 若 intent=modification：必须判断 modificationClear。仅当用户明确指定了「把什么改成什么」（具体修改对象+修改后的值）时填 true，否则填 false。若用户只说「想修改」「改一下」等未明确具体内容，填 false。modificationNewValue 仅当 modificationClear 为 true 时填写用户希望修改成的具体内容。
7. 若 intent=modification 且 modificationClear=true，填写 modificationTarget、modificationField 或 modificationValueStreamTarget；从【当前页面内容结构】中匹配最具体的字段名/环节名/阶段名。
8. 若涉及价值流图：从【当前页面内容结构】的价值流阶段与环节中精确匹配，modificationValueStreamLevel 填 step/stage/card，modificationValueStreamTarget 填匹配到的环节名或阶段名（必须与结构中出现的名称一致）
9. 若 intent=execute，填写 executeTaskId 和 executeTaskName。当用户说「重新进行需求逻辑构建」「重新构建需求逻辑」等时，intent=execute，executeTaskId=task3
10. summary：用一句话概括用户意图
11. 若无法明确推断，相关字段可填空字符串或合理默认值
12. 只返回 JSON，不要有 markdown 代码块包裹
```

### User Message 模板

```
用户输入：${text}

${context}
```

其中 `context` 由 `buildIntentExtractionContext()` 构建，包含【沟通历史】与【当前页面内容结构】。

## 3. 过程日志的提取逻辑

**函数**：`getCommunicationsByTask(createdAt)`  
**用途**：将聊天消息按任务分段，返回 `taskId -> communications`，供任务过程日志展示。

### 纳入规则

| 消息类型 | 纳入条件 |
|----------|----------|
| 用户消息 (role=user) | 纳入，但纯「确认」不纳入 |
| 意图卡片 (intentExtractionCard) | intent=query：不纳入；intent=discussion：不纳入（意图卡片本身不纳入，用户消息与系统回复已单独处理）；其他 intent：仅当 `confirmed=true` 时纳入 |
| 其他系统消息 (basicInfoCard、bmcCard、requirementLogicBlock、valueStreamCard) | 纳入 |
| 请教讨论的系统回复 (带 _taskId) | 纳入 |
| e2eFlowExtractStartBlock | 仅当 `confirmed=true` 时纳入 |
| e2eFlowJsonBlock | 仅当 `confirmed=true` 时纳入 |
| e2eFlowGeneratedLog | 纳入 |
| globalItGapStartBlock | 仅当 `confirmed=true` 时纳入 |
| globalItGapAnalysisCard | 仅当 `confirmed=true` 时纳入 |
| globalItGapAnalysisLog | 纳入 |
| localItGapStartBlock | 仅当 `confirmed=true` 时纳入 |
| localItGapSessionsBlock | 纳入（含 sessions 数据） |
| localItGapAnalysisCard | 仅当 `confirmed=true` 时纳入（含 analysisJson、llmMeta） |
| localItGapAnalysisLog | 纳入（含 analysisJson、llmMeta） |

### 讨论意图的特殊处理

- intent=discussion 且 confirmed 时：将用户消息移动到对应任务下，并插入一条 `speaker: '系统提炼'` 的提炼条目，content 为 `{ role, type: 'intentExtractionCard', content: '讨论请教', data, userText, timestamp }`。
- 其他已确认的意图卡片：以 `speaker: '系统大模型'` 纳入，content 为完整 payload（含 role、content、type、data、timestamp 等）。

### 不纳入的条目

- 查询意图 (query)：用户查询内容及系统返回的意图卡片均不纳入。
- 未确认的意图卡片：除 discussion 外，未确认的意图卡片及其触发的用户消息不纳入；discussion 会移除用户消息并插入系统提炼条目。

## 4. 过程日志的显示逻辑

### 4.1 展示位置

1. **任务追踪详情**：`renderTaskTrackingDetail()`，任务过程日志以列表形式展示。
2. **问题详情历史面板**：`renderProblemDetailHistory()`，任务过程日志以时间线形式展示。

### 4.2 用户意图提炼标题规则

所有对用户信息提炼的内容块（即 content 中 `type === 'intentExtractionCard'` 且存在 `data.intent` 的条目），标题统一显示为：

```
用户意图提炼：[意图展示名称]
```

其中 `[意图展示名称]` 由 `intentLabels[intent]` 映射得到，例如：

- `用户意图提炼：简单查询`
- `用户意图提炼：反馈修改意见`
- `用户意图提炼：执行操作`
- `用户意图提炼：讨论请教`

### 4.3 判断逻辑

```javascript
let titleLabel = c.speaker;
try {
  const parsed = typeof c.content === 'string' ? JSON.parse(c.content) : c.content;
  if (parsed?.type === 'intentExtractionCard' && parsed?.data?.intent != null) {
    const intentLabel = intentLabels[parsed.data.intent] || parsed.data.intent || '—';
    titleLabel = `用户意图提炼：${intentLabel}`;
  }
} catch (_) {}
```

无论原始 speaker 为「系统提炼」还是「系统大模型」，只要 content 为意图提炼卡片，均按上述规则显示标题。

### 4.4 ITGap 相关条目显示

| 类型 | 时间线标题 |
|------|------------|
| e2eFlowExtractStartBlock | 我先需要提取端到端流程绘制的 json 数据 |
| e2eFlowJsonBlock | 端到端流程 JSON 数据 |
| e2eFlowGeneratedLog | 已生成端到端流程 JSON 数据（详情含完整 JSON） |
| globalItGapStartBlock | 即将针对端到端流程开展全局 ITGap 分析 |
| globalItGapAnalysisCard | 全局 ITGap 分析（详情含 analysisJson） |
| globalItGapAnalysisLog | 已生成全局 ITGap 分析（详情含完整 analysisJson） |
| localItGapStartBlock | 即将生成每个环节的 ITGap 分析 session |
| localItGapSessionsBlock | 局部 ITGap 分析 Session（已为 N 个环节生成 session） |
| localItGapAnalysisCard | 局部 ITGap 分析：环节名（详情含 analysisJson、llmMeta） |
| localItGapAnalysisLog | 已生成环节「X」的局部 ITGap 分析（详情含 analysisJson、llmMeta） |

## 5. 任务列表参考

| taskId | 任务名称 | 归属阶段 |
|--------|----------|----------|
| task1 | 企业背景洞察 | 需求理解 |
| task2 | 商业画布加载 | 需求理解 |
| task3 | 需求逻辑构建 | 需求理解 |
| task4 | 绘制价值流 | 工作流对齐 |
| task5 | IT 现状标注 | 工作流对齐 |
| task6 | 痛点标注 | 工作流对齐 |
| task7 | 端到端流程绘制 | ITGap分析 |
| task8 | 全局 ITGap 分析 | ITGap分析 |
| task9 | 局部 ITGap 分析 | ITGap分析 |

**说明**：意图提炼的 `taskId` 仅来自 task1~task6。Task7~9 为 ITGap 阶段在沟通历史中展示的任务，其过程日志由确认型提示块触发写入。

### 5.1 ITGap 阶段与子步骤

| 大阶段 | 子步骤（按顺序） | 管理方式 |
|--------|------------------|----------|
| ITGap分析 (currentMajorStage=2) | 端到端流程绘制、全局 ITGap 分析、局部 ITGap 分析、流程节点生成、交互节点设计、集成节点设计、逻辑节点设计 | `itGapCompletedStages` 数组 |

**ITGap 入口提示块**（`maybeShowItGapStartBlock`）：

| 项目 | 说明 |
|------|------|
| 前置条件 | `workflowAlignCompletedStages` 包含 0、1、2（绘制价值流、IT现状标注、痛点标注均完成），且 `currentMajorStage >= 1` |
| 触发时机 | 痛点标注完成后，`maybeShowPainPointStartBlock` 内部会调用 `maybeShowItGapStartBlock` |
| 提示内容 | 「即将在现有价值流上开始 ITGap 分析」+「确认」按钮 |
| 确认后动作 | `currentMajorStage` 更新为 2，`itGapCompletedStages` 写入 `[0]`，工作区切换为 ITGap 分析页面（端到端流程绘制视图） |
| 消息类型 | `itGapStartBlock` |

**端到端流程提取流程**（`maybeShowE2eFlowExtractBlock`）：

- **触发条件**：进入全局 ITGap 分析（itGapCurrent === 1）且 Task7 的沟通历史为空
- **流程**：
  1. 系统提示「我先需要提取端到端流程绘制的 json 数据」+ [确认]（`e2eFlowExtractStartBlock`）
  2. 用户确认后，系统发送端到端流程 JSON 卡片 + [确认]（`e2eFlowJsonBlock`）
  3. 用户确认后，写入 `e2eFlowGeneratedLog`（含完整 valueStreamJson）
- **任务过程日志**：仅纳入系统提示块（e2eFlowExtractStartBlock、e2eFlowJsonBlock、e2eFlowGeneratedLog），用户「确认」不纳入

**全局 ITGap 分析流程**（`maybeShowGlobalItGapStartBlock`）：

- **触发条件**：进入全局 ITGap 分析（itGapCurrent === 1）、Task7 的沟通历史非空、尚未有 `globalItGapAnalysisJson`
- **流程**：
  1. 系统提示「即将针对端到端流程开展全局 ITGap 分析」+ [确认]（`globalItGapStartBlock`）
  2. 用户确认后，调用大模型生成分析，展示 JSON 卡片 + [确认] [重做]（`globalItGapAnalysisCard`，`structuredView=false`）
  3. 用户点击 [确认]：从 JSON 提取结构化内容，形成结构化卡片展示（含 7 个维度：全局架构失调诊断、决策协同断裂分析、数字化覆盖盲区、优先级建议矩阵、全局洞察结论、As-Is/To-Be、Top3 核心 IT 缺口），卡片带 [确认] [重做]（`structuredView=true`）
  4. 用户点击结构化卡片的 [确认]：写入工作区「全局ITGap 分析」卡片（含 Tab：全局ITGap 分析 / JSON），更新 `globalItGapAnalysisJson`，写入 `globalItGapAnalysisLog`（含 analysisJson）到 Task8 任务过程日志
- **删除联动**：删除聊天区任一全局 ITGap 相关消息（globalItGapStartBlock、globalItGapAnalysisCard、globalItGapAnalysisLog）时，同步清除工作区卡片及 Task8 任务过程日志中的全局 ITGap 分析内容

**局部 ITGap 分析流程**（`maybeShowLocalItGapStartBlock`、`forceShowLocalItGapStartBlock`）：

- **触发条件**：进入局部 ITGap 分析（itGapCurrent === 2）、已有全局 ITGap 分析、端到端流程有环节
- **流程**：
  1. 系统提示「即将生成每个环节的 ITGap 分析 session」+ [确认]（`localItGapStartBlock`）
  2. 用户确认后，系统根据端到端流程生成各环节 session（不调用大模型），展示 session 列表卡片 + [确认] [继续]（`localItGapSessionsBlock`）
  3. 用户点击 [确认] 或 [继续]：直接开始逐环节调用大模型分析，无需二次确认
  4. 每完成一个环节：展示 `localItGapAnalysisCard`（含 LLM 元信息）、`localItGapAnalysisLog`；更新工作区对应 session 卡片（待分析→已分析✅）；写入 Task9 任务过程日志（含 analysisJson、llmMeta）
  5. 自动继续下一环节，直至全部完成
- **继续按钮**：session 列表卡片在 [确认] 右侧有 [继续] 按钮，存在未完成分析时可用，用于从中断处继续执行

## 6. 过程日志中不纳入的条目（补充）

| 消息类型/内容 | 说明 |
|---------------|------|
| `itGapStartBlock` | ITGap 入口提示块，`shouldIncludeInCommunicationHistory` 未纳入，故不进入任务过程日志 |
| 用户纯「确认」 | 用户点击确认按钮产生的 `content: '确认'` 消息不纳入，避免过程日志充斥确认记录 |
